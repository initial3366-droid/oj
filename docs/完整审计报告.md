## 二、安全审查 ⚠️ 

### 2.1 严重问题

#### ❌ LocalJudgeService 直接执行用户代码
**文件**: `LocalJudgeService.java:369`
```java
Process process = new ProcessBuilder(command).directory(workDir.toFile()).start();
```

**风险**:
- 用户代码直接在主服务器执行
- 无沙箱隔离
- 可能导致：服务器资源耗尽、文件系统访问、反向shell

**建议**:
1. 生产环境禁用 LocalJudgeService
2. 只使用 DOMjudge 进行判题
3. 如需本地判题，必须使用 Docker 沙箱

#### ✅ 权限控制 - 已修复
- ✅ ContestRankController 添加了 `@PreAuthorize` 注解
- ✅ SecurityConfig 添加了排名 API 的访问规则
- ⚠️  部分 Admin Controller 仍需检查

### 2.2 中等风险

#### ⚠️  临时文件清理
`LocalJudgeService.java` 的 `deleteDir()` 方法捕获了所有异常，可能导致临时文件残留。

#### ⚠️  SQL 注入防护
- ✅ 使用 MyBatis-Plus 的 LambdaQueryWrapper，安全
- ✅ 没有发现原生 SQL 拼接

### 2.3 XSS 防护
- ✅ 后端使用 Jackson 序列化，自动转义
- ✅ 前端使用 React，默认转义
- ✅ KaTeX 数学渲染使用了正确的配置

## 六、判题系统审查 ⚠️

### 6.1 判题架构
系统支持两种判题方式：
1. **DOMjudge** (推荐) - 外部判题系统，通过 API 集成
2. **LocalJudgeService** (危险) - 本地判题服务

### 6.2 LocalJudgeService 问题
```java
// 直接在主服务器执行用户代码
Process process = new ProcessBuilder(command)
    .directory(workDir.toFile())
    .start();
```

**无任何隔离措施**:
- ❌ 无 Docker 容器
- ❌ 无 chroot 监牢
- ❌ 无 seccomp 系统调用过滤
- ❌ 无网络隔离
- ❌ 无资源限制（除了简单的超时）

### 6.3 建议
1. **生产环境**: 完全禁用 LocalJudgeService
2. **开发环境**: 仅用于快速测试，并添加警告
3. **长期方案**: 实现基于 Docker 的判题沙箱
