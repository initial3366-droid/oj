#### 资源级权限系统

**架构设计**：
```
security/policy/
├── Permission.java              # 权限枚举（10种权限类型）
├── AccessPolicy.java            # 权限策略基类
├── ProblemAccessPolicy.java     # 题目权限策略
├── ContestAccessPolicy.java     # 比赛权限策略
├── PracticeAccessPolicy.java    # 练习权限策略
├── SubmissionAccessPolicy.java  # 提交权限策略
└── ScoreboardAccessPolicy.java  # 榜单权限策略
```

**支持的权限动作**：
- VIEW - 查看资源
- CREATE - 创建资源
- UPDATE - 修改资源
- DELETE - 删除资源
- SUBMIT - 提交代码
- VIEW_CODE - 查看代码
- VIEW_HIDDEN_CASE - 查看隐藏测试用例
- REJUDGE - 重新判题
- MANAGE_REGISTRATION - 管理报名
- MANAGE_SCOREBOARD - 管理榜单

**核心原则**：
1. 不仅基于角色（RBAC），更注重资源所有权和上下文
2. 默认拒绝，显式授权
3. 代码和私有资源访问必须严格校验
4. 支持扩展新权限类型

**使用示例**：
```java
@Service
public class ProblemService {
    private final ProblemAccessPolicy problemAccessPolicy;
    
    public ProblemVO detail(long id) {
        Problem problem = problemMapper.selectById(id);
        AuthUser user = CurrentUser.get();
        
        if (!problemAccessPolicy.can(user, Permission.VIEW, problem)) {
            throw new BizException(ErrorCode.NOT_FOUND, "题目不存在");
        }
        
        return toVO(problem);
    }
}
```

```sql
SET @sql = IF(

        SELECT 1 FROM information_schema.COLUMNS

          AND COLUMN_NAME = 'scoring_mode'
    ),
    'ALTER TABLE contests ADD COLUMN scoring_mode VARCHAR(16) NOT NULL DEFAULT ''ACM'' ...',

PREPARE stmt FROM @sql; 
EXECUTE stmt; 
DEALLOCATE PREPARE stmt;
```

## 四、技术债务

### 4.1 当前技术债务

#### 前端
1. 自定义 heroui-adapter 维护成本高
   - **影响**：升级困难，可能与上游不兼容
   - **建议**：迁移到真正的 HeroUI 或其他成熟 UI 库
   
2. 打包体积过大（673 kB）
   - **影响**：首屏加载慢
   - **建议**：代码分割，按路由懒加载
   
3. 缺少单元测试
   - **影响**：重构风险高
   - **建议**：使用 Vitest 添加测试覆盖

#### 后端
1. Service 层方法过长
   - **影响**：可读性差，难以维护
   - **建议**：拆分大方法，提取私有辅助方法
   
2. 缺少集成测试
   - **影响**：回归测试依赖手动验证
   - **建议**：使用 @SpringBootTest 添加集成测试
   
3. 部分接口缺少 API 文档注释
   - **影响**：前后端对接成本高
   - **建议**：添加 Swagger/OpenAPI 注解
   
4. 判题逻辑耦合度高
   - **影响**：切换判题服务困难
   - **建议**：抽象 JudgeAdapter 接口

#### 数据库
1. 缺少数据库连接池监控
   - **影响**：无法及时发现连接泄漏
   - **建议**：集成 HikariCP 监控
   
2. 未启用慢查询日志
   - **影响**：性能问题难以排查
   - **建议**：配置 MySQL 慢查询日志
   
3. 缺少定期备份方案
   - **影响**：数据丢失风险
   - **建议**：配置定时备份脚本

## 八、经验总结

### 8.1 成功经验

1. **安全优先原则**
   - 先修复高危漏洞，再优化性能
   - 默认拒绝，显式授权

2. **统一规范**
   - API 响应格式统一降低对接成本
   - 错误码枚举提高可维护性

3. **分层架构**
   - Policy 层负责权限判断逻辑
   - Service 层负责业务逻辑和数据查询
   - Controller 层只做参数校验和调用

4. **渐进式重构**
   - 先修复编译错误
   - 再统一响应结构
   - 最后重构权限系统

### 8.2 经验教训

1. **依赖管理**
   - 避免使用 "latest" 版本
   - 定期更新但锁定版本号

2. **数据库迁移**
   - 使用条件添加避免冲突
   - 版本号不要跳跃过大

3. **权限设计**
   - 不能只依赖角色判断
   - 必须考虑资源所有权

4. **文档先行**
   - 先设计架构文档
   - 再编写代码实现
