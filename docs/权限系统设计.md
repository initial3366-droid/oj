## 一、权限系统架构

### 1.1 设计原则

1. **资源级权限控制**：不仅基于角色（RBAC），更注重资源所有权和上下文
2. **最小权限原则**：默认拒绝，显式授权
3. **安全优先**：涉及代码、私有资源的访问必须严格校验
4. **可扩展性**：支持新增权限类型和资源类型

### 1.2 核心组件

```
security/
├── AuthUser.java                    # 认证用户信息
├── CurrentUser.java                 # 当前用户工具类
├── policy/
│   ├── Permission.java              # 权限枚举
│   ├── AccessPolicy.java            # 权限策略基类
│   ├── ProblemAccessPolicy.java     # 题目权限策略
│   ├── ContestAccessPolicy.java     # 比赛权限策略
│   ├── PracticeAccessPolicy.java    # 练习权限策略
│   ├── SubmissionAccessPolicy.java  # 提交权限策略
│   └── ScoreboardAccessPolicy.java  # 榜单权限策略
```

## 二、角色定义

### 2.1 角色层级

| 角色 | 英文名 | 权限范围 | 说明 |
|------|--------|----------|------|
| 超级管理员 | SUPER_ADMIN | 全局所有资源 | 拥有系统所有权限 |
| 教师 | TEACHER | 自己创建的资源 + 班级学生 | 可管理班级、创建比赛/练习/题目 |
| 社团管理员 | CLUB_ADMIN | 社团内资源 | 可管理社团、创建社团比赛/练习 |
| 学生 | STUDENT | 公开资源 + 授权资源 | 参与比赛、练习，提交代码 |
| 访客 | GUEST | 公开资源 | 仅查看权限 |

### 2.2 账号类型

- **前台账号** (User)：学生、教师、社团管理员
- **后台账号** (AdminUser)：超级管理员，不能参与比赛/练习

### 3.1 Permission 枚举

```java
public enum Permission {
    VIEW,                    // 查看资源
    CREATE,                  // 创建资源
    UPDATE,                  // 修改资源
    DELETE,                  // 删除资源
    SUBMIT,                  // 提交代码
    VIEW_CODE,               // 查看代码
    VIEW_HIDDEN_CASE,        // 查看隐藏测试用例
```

## 五、使用指南

### 5.1 在 Service 层使用 Policy

```java
@Service
public class ProblemService {
    private final ProblemAccessPolicy problemAccessPolicy;
    
    public ProblemVO detail(long id) {
        Problem problem = problemMapper.selectById(id);
        AuthUser user = CurrentUser.get();
        
        // 检查查看权限
        if (!problemAccessPolicy.can(user, Permission.VIEW, problem)) {
            throw new BizException(ErrorCode.NOT_FOUND, "题目不存在");
        }
        
        return toVO(problem);
    }
    
    public void update(long id, ProblemUpdateRequest request) {
        Problem problem = problemMapper.selectById(id);
        AuthUser user = CurrentUser.required();
        
        // 检查修改权限
        if (!problemAccessPolicy.can(user, Permission.UPDATE, problem)) {
            throw new BizException(ErrorCode.FORBIDDEN);
        }
        
        // 执行更新逻辑
    }
}
```

### 5.2 在 Controller 层使用 Policy（可选）

虽然推荐在Service层进行权限检查，但也可以在Controller层做预检查：

```java
@RestController
@RequestMapping("/api/v1/problems")
public class ProblemController {
    private final ProblemService problemService;
    private final ProblemAccessPolicy problemAccessPolicy;
    
    @PostMapping
    public ApiResponse<Void> create(@RequestBody ProblemCreateRequest request) {
        AuthUser user = CurrentUser.required();
        
        // Controller层预检查
        if (!problemAccessPolicy.can(user, Permission.CREATE, null)) {
            throw new BizException(ErrorCode.FORBIDDEN, "无权创建题目");
        }
        
        problemService.create(request);
        return ApiResponse.ok();
    }
}
```

### 5.3 特殊场景处理

#### 场景1：需要数据库查询的权限判断

Policy只能基于已加载的资源判断，涉及关联查询的权限需在Service层完成：

```java
// ContestAccessPolicy 判断 audience 是否匹配
public boolean canView(AuthUser user, Contest contest) {
    if ("ALL".equals(contest.audience)) {
        return true;
    }
    // 对于 CLASS/CLUB，返回 false，由 Service 层进一步检查
    return false;
}

// ContestService 完成完整权限检查
public ContestVO detail(long id) {
    Contest contest = contestMapper.selectById(id);
    AuthUser user = CurrentUser.get();
    
    // 基础权限检查
    if (!contestAccessPolicy.can(user, Permission.VIEW, contest)) {
        // 检查是否是 CLASS/CLUB 成员
        if ("CLASS".equals(contest.audience)) {
            boolean isMember = checkClassMember(contest.audienceId, user.id());
            if (!isMember) {
                throw new BizException(ErrorCode.FORBIDDEN);
            }
        } else if ("CLUB".equals(contest.audience)) {
            boolean isMember = checkClubMember(contest.audienceId, user.id());
            if (!isMember) {
                throw new BizException(ErrorCode.FORBIDDEN);
            }
        } else {
            throw new BizException(ErrorCode.NOT_FOUND, "比赛不存在");
        }
    }
    
    return toVO(contest);
}
```

#### 场景2：组合权限判断

```java
// 查看比赛题目详情需要组合判断
public ProblemVO contestProblemDetail(long contestId, long problemId) {
```

## 八、迁移指南

### 8.1 现有代码迁移步骤

1. **识别权限检查点**：在Service层找到所有权限相关的if判断
2. **引入Policy**：注入对应的AccessPolicy
3. **替换硬编码判断**：用`policy.can()`替换原有if条件
4. **测试验证**：确保迁移后行为一致

### 8.2 迁移示例

**迁移前**：
```java
public ProblemVO detail(long id) {
    Problem problem = problemMapper.selectById(id);
    if (!problem.isPublic) {
        AuthUser user = CurrentUser.get();
        if (user == null) {
            throw new BizException(404, "题目不存在");
        }
        if (!"SUPER_ADMIN".equals(user.role()) && !problem.ownerId.equals(user.id())) {
            throw new BizException(404, "题目不存在");
        }
    }
    return toVO(problem);
}
```

**迁移后**：
```java
public ProblemVO detail(long id) {
    Problem problem = problemMapper.selectById(id);
    AuthUser user = CurrentUser.get();
    
    if (!problemAccessPolicy.can(user, Permission.VIEW, problem)) {
        throw new BizException(ErrorCode.NOT_FOUND, "题目不存在");
    }
    
    return toVO(problem);
}
```

## 九、常见问题

### Q1: Policy层和Service层的权限检查如何分工？

**A**: 
- **Policy层**：基于资源本身的权限判断（如是否公开、是否创建者、角色是否匹配）
- **Service层**：需要额外数据库查询的权限判断（如成员关系、报名状态）

### Q2: 为什么不用Spring Security的@PreAuthorize？

**A**: `@PreAuthorize`基于SpEL表达式，难以表达复杂的资源级权限逻辑。Policy模式更灵活、可测试性更强。

### Q3: Policy会影响性能吗？

**A**: Policy本身是纯内存判断，性能影响极小。如果涉及数据库查询，建议在Service层缓存结果。

### Q4: 如何扩展新的Permission？

**A**: 
1. 在`Permission`枚举中添加新的权限
2. 在对应Policy的`can()`方法中添加case分支
3. 实现具体的权限判断逻辑
