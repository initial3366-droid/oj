## 🔴 发现的严重问题

### 1. **重复表结构：submission_cases vs submission_case_results**

**问题**:
- V1 创建了 `submission_cases` 表
- V15 创建了 `submission_case_results` 表
- 两张表功能完全相同，造成混乱

**影响**:
- 数据可能分散在两张表
- 代码不知道该用哪张表
- 浪费存储空间

### 2. **contest_problems 表主键冲突**

**问题**:
- V1 创建时使用复合主键 `(contest_id, problem_id)`
- V13 尝试添加 `id` 列作为主键
- 可能导致迁移失败

**现状分析**:
```sql
-- V1: 复合主键
PRIMARY KEY (contest_id, problem_id)

-- V13: 尝试添加自增 id
ALTER TABLE contest_problems ADD COLUMN id BIGINT NOT NULL AUTO_INCREMENT UNIQUE FIRST
```

### 3. **废弃表未完全清理**

**问题**:
- `_deprecated_contest_standings` 表被重命名但仍占用空间
- `submission_cases` 被标记为旧版但未迁移数据

## ✅ 修复措施

### 修复 1: 统一提交测试点结果表

**策略**: 保留 `submission_case_results`，废弃 `submission_cases`

**新增迁移**: `V21__unify_submission_case_tables.sql`

```sql
-- V21: 统一提交测试点结果表

-- 1. 迁移数据从 submission_cases 到 submission_case_results
INSERT INTO submission_case_results (
    submission_id, case_no, status, time_ms, memory_kb, created_at
)
SELECT 
    submission_id, case_no, status, time_ms, memory_kb, NOW()
FROM submission_cases
WHERE NOT EXISTS (
    SELECT 1 FROM submission_case_results scr
    WHERE scr.submission_id = submission_cases.submission_id
    AND scr.case_no = submission_cases.case_no
);

-- 2. 删除旧表
DROP TABLE IF EXISTS submission_cases;

-- 3. 添加注释
ALTER TABLE submission_case_results 
COMMENT '提交测试点结果表（统一表，替代旧的 submission_cases）';
```

### 修复 2: contest_problems 表结构说明

**V13 的处理是正确的**:
- 使用动态 SQL 检查列是否存在
- 安全地添加 `id` 列
- 保持向后兼容

**验证 V13 的安全性**:
```sql
SET @sql = IF(
    NOT EXISTS (
        SELECT 1 FROM information_schema.COLUMNS
        WHERE TABLE_NAME = 'contest_problems'
        AND COLUMN_NAME = 'id'
    ),
    'ALTER TABLE contest_problems ADD COLUMN id ...',
    'SELECT 1'  -- 已存在则跳过
);
```

**结论**: ✅ V13 已经做了幂等性保护，无需修复

### 修复 3: 清理废弃表

**新增迁移**: `V22__cleanup_deprecated_tables.sql`

