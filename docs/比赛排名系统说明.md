# Contest Ranking System - ACM/OI 双模式排名系统

## 概述

本系统实现了完整的 ACM 和 OI 双模式比赛排名功能，支持实时排名、封榜快照、全量重建等核心功能。

## 排名规则

### ACM 模式

**排序规则**（优先级从高到低）：
1. 解决题数（solved_count）降序
2. 总罚时（penalty_time）升序
3. 最后 AC 时间（last_ac_time）升序

**罚时计算**：
```
penalty_time = solve_time_minutes + wrong_attempts × penalty_minutes
```

- `solve_time_minutes`：从比赛开始到 AC 的分钟数
- `wrong_attempts`：AC 前的错误提交次数（WA/TLE/MLE/RE，CE 可选）
- `penalty_minutes`：每次错误的罚时分钟数（默认 20）

**状态说明**：
- 题目 AC 后不再接受新的罚时
- CE（编译错误）是否计入罚时由 `count_ce_as_penalty` 控制
- 未 AC 的题目只显示错误次数，不计入罚时

### OI 模式

**排序规则**（优先级从高到低）：
1. 总分（total_score）降序
2. 满分题数（solved_count）降序
3. 最后得分更新时间（last_score_update_time）升序

**计分规则**：
```
total_score = sum(best_score per problem)
```

- 每道题取历史最高分
- 满分题数：`best_score >= full_score` 的题目数量
- 支持部分分，每个测试点可以有不同的分值

## 数据库设计

### 核心表

#### contest_acm_rank_cache（ACM 总榜缓存）
```sql
- contest_id, participant_id (联合主键)
- rank_no: 排名
- solved_count: 解决题数
- penalty_time: 总罚时
- submission_count: 提交次数
- last_ac_time: 最后 AC 时间
```

#### contest_acm_rank_problems（ACM 单题状态）
```sql
- contest_id, participant_id, contest_problem_id (联合主键)
- is_solved: 是否 AC
- wrong_attempts: 错误次数
- solve_time_minutes: AC 时间（分钟）
- first_ac_submission_id: 首次 AC 提交 ID
```

#### contest_oi_rank_cache（OI 总榜缓存）
```sql
- contest_id, participant_id (联合主键)
- rank_no: 排名
- total_score: 总分
- solved_count: 满分题数
- submission_count: 提交次数
- last_score_update_time: 最后得分更新时间
```

#### contest_oi_rank_problems（OI 单题状态）
```sql
- contest_id, participant_id, contest_problem_id (联合主键)
- best_score: 历史最高分
- full_score: 满分
- submission_count: 提交次数
- best_submission_id: 最高分提交 ID
- first_full_score_time: 首次满分时间
```

#### contest_participants（参赛者）
```sql
- id (主键)
- contest_id, user_id
- nickname: 显示昵称
- identity_type, identity_id: 身份类型（班级/社团）
- class_id: 班级 ID（用于显示）
- status: 参赛状态
```

#### submission_case_results（测试点结果）
```sql
- submission_id, case_no (联合主键)
- subtask_no: 子任务编号
- status: 测试点状态
- score, max_score: 得分和满分
- time_used, memory_used: 资源使用
```

#### contest_scoreboard_snapshots（榜单快照）
```sql
- id (主键)
- contest_id
- snapshot_type: 'freeze'（封榜）或 'final'（最终榜）
- ranking_data: JSON 格式的完整榜单数据
- snapshot_time: 快照时间
```

### 索引策略

所有缓存表和单题表都在 `(contest_id, participant_id)` 上建立唯一索引，保证查询和更新效率。

## 后端实现

### 核心服务

#### ContestAcmRankService
- `updateRankAfterJudge(Submission)`: 判题后更新排名（幂等）
- `rebuildRank(Long contestId)`: 全量重建排名
- `getRankList(Long contestId)`: 获取完整榜单

**幂等性保证**：
```java
// 已 AC 的题目不再处理
if (Boolean.TRUE.equals(problemRank.isSolved)) {
    return;
}
```

#### ContestOiRankService
- `updateRankAfterJudge(Submission)`: 判题后更新排名（幂等）
- `rebuildRank(Long contestId)`: 全量重建排名
- `getRankList(Long contestId)`: 获取完整榜单

**幂等性保证**：
```java
// 只在分数提高时更新
boolean scoreImproved = currentScore > oldBestScore;
if (scoreImproved) {
    int scoreDelta = currentScore - oldBestScore;
    // 使用增量更新总分
}
```

#### JudgeCallbackService
判题结果回调统一入口：
1. 更新 `submissions` 表（status, score, time_used 等）
2. 批量保存 `submission_case_results`
3. 根据比赛类型分发到 ACM/OI 排名服务
4. 整个流程使用 `@Transactional` 保证原子性

#### ContestScoreboardSnapshotService
榜单快照管理：
- `createSnapshot(contestId, type)`: 创建快照
- `getSnapshot(contestId, type)`: 获取快照
- `deleteSnapshot(contestId, type)`: 删除快照

### REST API

#### 获取排名
- `GET /api/contests/{contestId}/rank/acm` - 获取 ACM 榜单
- `GET /api/contests/{contestId}/rank/oi` - 获取 OI 榜单

#### 管理操作（需要管理员权限）
- `POST /api/contests/{contestId}/rank/acm/rebuild` - 重建 ACM 榜单
- `POST /api/contests/{contestId}/rank/oi/rebuild` - 重建 OI 榜单
- `POST /api/contests/{contestId}/scoreboard/snapshot?type={freeze|final}` - 创建快照
- `GET /api/contests/{contestId}/scoreboard/snapshot/{type}` - 获取快照
- `DELETE /api/contests/{contestId}/scoreboard/snapshot/{type}` - 删除快照

## 使用流程

### 创建比赛
```sql
INSERT INTO contests (title, type, scoring_mode, penalty_minutes, count_ce_as_penalty, ...)
VALUES ('比赛名称', 'ACM', 'ACM', 20, false, ...);
```

- `type`: 'ACM' 或 'OI'（向后兼容）
- `scoring_mode`: 'ACM' 或 'OI'（新字段，优先级高于 type）
- `penalty_minutes`: ACM 模式每次错误的罚时（默认 20）
- `count_ce_as_penalty`: CE 是否计入罚时（默认 false）

### 封榜流程
1. 设置封榜时间：`UPDATE contests SET freeze_time = '2024-01-01 12:00:00' WHERE id = 1;`
2. 创建封榜快照：`POST /api/contests/1/scoreboard/snapshot?type=freeze`
3. 比赛结束后创建最终快照：`POST /api/contests/1/scoreboard/snapshot?type=final`
4. 前端可选择显示：实时榜 / 封榜快照 / 最终榜

### 排名重建
适用场景：
- 数据迁移后
- 排名算法调整后
- 缓存数据损坏

操作：
```
POST /api/contests/{contestId}/rank/acm/rebuild
或
POST /api/contests/{contestId}/rank/oi/rebuild
```

系统会从 `submissions` 表按时间顺序重新处理所有提交，完全重建排名缓存。

## 事务和一致性

### 判题回调事务
```java
@Transactional(rollbackFor = Exception.class)
public void handleJudgeResult(JudgeResultCallbackRequest request)
```

保证：
- Submission 更新
- SubmissionCaseResult 批量插入
- 排名更新

全部成功或全部回滚。

### 幂等性
所有排名更新方法都是幂等的：
- ACM：已 AC 的题目不再处理
- OI：只在分数提高时更新，使用增量计算

这意味着：
- 判题系统可以安全重试
- 全量重建可以多次执行
- 不会出现重复计分或罚时

## 性能优化

1. **缓存设计**：`rank_cache` 表避免每次查询都聚合 `submissions`
2. **增量更新**：只在分数改变时更新缓存，不做无效写入
3. **批量查询**：`getRankList` 使用 `selectBatchIds` 批量获取用户信息
4. **索引覆盖**：所有查询都走索引，避免全表扫描

## 监控和诊断

### 检查排名一致性
```sql
-- ACM 模式：对比缓存和实际提交
SELECT 
    c.participant_id,
    c.solved_count AS cache_solved,
    COUNT(DISTINCT p.contest_problem_id) AS actual_solved
FROM contest_acm_rank_cache c
LEFT JOIN contest_acm_rank_problems p ON p.contest_id = c.contest_id 
    AND p.participant_id = c.participant_id AND p.is_solved = true
WHERE c.contest_id = ?
GROUP BY c.participant_id
HAVING cache_solved != actual_solved;
```

### 检查参赛者绑定
```sql
-- 查看未绑定 participant_id 的提交
SELECT COUNT(*) 
FROM submissions 
WHERE contest_id = ? AND participant_id IS NULL;
```

如果有未绑定的提交，检查触发器是否正常工作。

## 故障排查

### 排名不更新
1. 检查 `submissions.participant_id` 是否正确
2. 检查 `contest.scoring_mode` 是否与调用的服务匹配
3. 查看日志中是否有异常回滚

### 罚时计算错误
1. 确认 `contest.penalty_minutes` 设置
2. 确认 `contest.count_ce_as_penalty` 设置
3. 检查 `contest_acm_rank_problems.wrong_attempts` 是否准确

### 分数不一致
1. 运行全量重建：`POST /api/contests/{id}/rank/oi/rebuild`
2. 对比 `submission_case_results` 和 `contest_oi_rank_problems.best_score`
3. 检查是否有并发写入导致的脏数据
