#### 1. **com/qoj/config/QojProperties.java**
**修改内容**: 添加 @PostConstruct 验证方法

**核心改动**:
```java
@PostConstruct
public void validate() {
    int secretLength = jwt.secret.getBytes(StandardCharsets.UTF_8).length;
    String activeProfile = System.getProperty("spring.profiles.active", "dev");
    boolean isProduction = "prod".equalsIgnoreCase(activeProfile) || 
                          "production".equalsIgnoreCase(activeProfile);
    
    // 生产环境强制要求
    if (secretLength < 32) {
        if (isProduction) {
            throw new IllegalStateException("【生产环境启动失败】JWT_SECRET 长度不足");
        }
    }
    
    // 检查默认值
    if (jwt.secret.contains("change-this") || jwt.secret.contains("default")) {
        if (isProduction) {
            throw new IllegalStateException("【生产环境启动失败】JWT_SECRET 使用了默认值");
        }
    }
}
```

**安全效果**:
- 开发环境：控制台警告 ⚠️
- 生产环境：启动失败，强制修改 ❌

**修改内容**: Token 轮换支持

**核心改动**:
```java
// 添加 familyId 到 TokenPair record
public record TokenPair(String accessToken, String refreshToken, String familyId) {}

// issueTokens 生成 familyId
public TokenPair issueTokens(AuthUser user) {
    String familyId = UUID.randomUUID().toString();
    return new TokenPair(
        issueToken(user, accessTokenTtl, "access", null),
        issueToken(user, refreshTokenTtl, "refresh", familyId),
        familyId
    );
}

// 添加 familyId 到 refresh token claims
if (familyId != null && "refresh".equals(tokenType)) {
```

#### 3. **com/qoj/module/auth/service/AuthService.java**
**修改内容**: 核心安全逻辑重构（最大改动）

**3.1 login() 方法**
```java
// 保存 refresh token family 到 Redis
redisTemplate.opsForValue().set(
    RedisKeys.refreshTokenFamily(pair.familyId()),
    pair.refreshToken(),
    Duration.ofDays(7)
);
```

**3.2 refresh() 方法 - Token 轮换核心**
```java
// 1. 检查 refresh token 是否在黑名单
if (Boolean.TRUE.equals(redisTemplate.hasKey(RedisKeys.refreshTokenBlacklist(jti)))) {
    throw new BizException(401, "Refresh Token 已失效");
}

// 2. 检查 family 并检测重用
String storedToken = redisTemplate.opsForValue().get(RedisKeys.refreshTokenFamily(familyId));
if (!request.refreshToken().equals(storedToken)) {
    // 检测到 refresh token 重用，撤销整个 family
    redisTemplate.delete(RedisKeys.refreshTokenFamily(familyId));
    throw new BizException(401, "检测到 Token 重用，已撤销所有 Token");
}

// 3. 将旧的 refresh token 加入黑名单
redisTemplate.opsForValue().set(
    RedisKeys.refreshTokenBlacklist(jti),
    "1",
    Duration.ofSeconds(jwtService.remainingSeconds(claims))
);

// 4. 签发新的 tokens（轮换）
String newAccessToken = jwtService.issueAccessToken(authUser);
String newRefreshToken = jwtService.issueRefreshToken(authUser, familyId);

// 5. 更新 family 中的最新 token
redisTemplate.opsForValue().set(
    RedisKeys.refreshTokenFamily(familyId),
    newRefreshToken,
    Duration.ofDays(7)
);
```

**安全机制**:
- **Token 轮换**: 每次刷新都签发新 Token，旧 Token 立即失效
- **重用检测**: 如果攻击者使用被窃取的旧 Token，系统检测到后撤销整个家族
- **双层防护**: 黑名单 + 家族跟踪

**3.3 logout() 方法 - 修复安全漏洞**
```java
// 将 access token 加入黑名单
redisTemplate.opsForValue().set(
    RedisKeys.tokenBlacklist(jti),
    "1",
    Duration.ofSeconds(jwtService.remainingSeconds(claims))
);

// 如果是 refresh token，撤销整个 family
if ("refresh".equals(tokenType)) {
    String familyId = claims.get("familyId", String.class);
    if (familyId != null) {
        redisTemplate.delete(RedisKeys.refreshTokenFamily(familyId));
    }
    
    // 也将 refresh token 自己加入黑名单
    redisTemplate.opsForValue().set(
        RedisKeys.refreshTokenBlacklist(jti),
        "1",
        Duration.ofSeconds(jwtService.remainingSeconds(claims))
    );
}
```

**修复前问题**: 只撤销 Access Token，Refresh Token 仍可用
**修复后**: 同时撤销 Access Token、Refresh Token 和整个 Token 家族

### Refresh Token 重用检测机制

**场景**: 攻击者窃取了 Refresh Token

```
时间线：
T0: 用户正常登录，获得 RT1（familyId=abc）
T1: 攻击者窃取 RT1
T2: 攻击者使用 RT1 刷新，获得 RT2
    → Redis familyId:abc = RT2
    → RT1 进入黑名单
T3: 真实用户使用 RT1 刷新（尝试）
    → 系统发现 RT1 != familyId:abc（RT2）
    → 检测到重用！
    → 撤销整个 family（删除 familyId:abc）
    → 返回 401，所有 Token 失效
```

**安全效果**: 
- 攻击者窃取 Token 后首次使用会成功
- 真实用户第二次使用会触发检测
- 系统撤销所有相关 Token，强制重新登录

**估算**: 1000 活跃用户

### 核心技术点

**1. Token 轮换 (Token Rotation)**
- 每次刷新生成新的 Refresh Token
- 旧 Token 立即失效（黑名单）
- 防止 Token 长期有效带来的风险

**2. Token 重用检测 (Reuse Detection)**
- 基于 familyId 的 Token 家族管理
- 检测到旧 Token 重用时撤销整个家族
- 有效防御 Token 窃取攻击

**3. 双层黑名单 (Dual Blacklist)**
- Access Token 黑名单：快速撤销访问权限
- Refresh Token 黑名单：防止刷新操作
- TTL 自动过期：无需手动清理

**4. 生产环境保护 (Production Safeguards)**
- 启动时强制验证 JWT_SECRET
- 禁止使用默认值
- 配置文件隔离（dev/prod）

## ⚠️ 已知限制和后续优化

### 当前限制

**1. localStorage XSS 风险**
- **现状**: Token 存储在 localStorage，存在 XSS 攻击风险
- **短期缓解**: Access Token 短有效期（15分钟）
- **长期方案**: 迁移到 HttpOnly Cookie（详见安全指南）

**2. 缺少 CSRF 防护**
- **现状**: 使用 Bearer Token，不受 CSRF 影响
- **影响**: 迁移到 Cookie 后需要实施 CSRF 防护
- **方案**: Double Submit Cookie 或 SameSite Cookie

**3. 缺少 CSP 配置**
- **现状**: 未配置 Content-Security-Policy 响应头
- **影响**: 无法限制内联脚本执行
- **方案**: 配置 CSP Filter（代码示例见安全指南）
