## 发现的问题

### 1. **严重问题：双榜单系统共存** ⚠️

**问题描述**:
- **旧系统**（V1 迁移）：`contest_standings` 表，由 `JudgePollingService.updateContestStanding()` 更新
- **新系统**（V15 迁移）：`contest_acm_rank_cache` 和 `contest_oi_rank_cache` 表，由 `ContestAcmRankService` 和 `ContestOiRankService` 维护
- 两套系统同时存在，造成数据不一致和维护困难

**旧系统代码位置**:
```java
// JudgePollingService.java:7-9

// JudgePollingService.java:168-194

    // 使用 contest_standings 表
    ContestStanding standing = standingMapper.selectOne(...);
    // 简单的 AC 计数逻辑，不支持 OI 赛制
}
```

**新系统代码位置**:
```java
// ContestAcmRankService.java:53-133
public void updateRankAfterJudge(Submission submission) {
    // 幂等的 ACM 排名更新
    // 支持罚时、错误次数、封榜
}

// ContestOiRankService.java:51-119

    // 幂等的 OI 排名更新
    // 支持分数、满分题数

```

**影响**:
- 数据不一致：旧系统和新系统可能产生不同的榜单结果
- 维护困难：修改榜单逻辑需要同时修改两套系统
- 功能缺失：旧系统不支持 OI 赛制、封榜、快照等功能
- 资源浪费：维护两套表和逻辑

### 2. **中等问题：API 路径混乱**

**问题描述**:
- ContestRankController 使用 `/api/contests` 路径（不符合规范）
- 管理操作混在用户操作中（rebuild, snapshot create/delete）
- 缺少统一的榜单查询接口

**现有路径**:
```java

public class AdminContestRankController {
    /**
     * 重建榜单（管理员）
     * POST /api/admin/v1/contests/{contestId}/rank/rebuild?mode=ACM/OI
     */
    @PostMapping("/{contestId}/rank/rebuild")
    public ApiResponse<Void> rebuildRank(
        @PathVariable Long contestId,
        @RequestParam String mode
    ) {
        // 权限检查
        Contest contest = contestMapper.selectById(contestId);
        AuthUser user = CurrentUser.required();
        if (!contestAccessPolicy.can(user, Permission.UPDATE, contest)) {
            throw new BizException(ErrorCode.FORBIDDEN, "无权重建该比赛榜单");

            acmRankService.rebuildRank(contestId);
```

**修改文件**: `ContestRankController.java`

```java
@RestController
@RequestMapping("/api/v1/contests")  // ✅ 修改路径
public class ContestRankController {
    /**
     * 统一榜单接口
     * GET /api/v1/contests/{contestId}/rank?mode=ACM/OI
     */
    @GetMapping("/{contestId}/rank")
    public ApiResponse<?> getRank(
        @PathVariable Long contestId,
        @RequestParam(required = false) String mode
    ) {
        Contest contest = contestMapper.selectById(contestId);
        if (contest == null) {
            throw new BizException(ErrorCode.NOT_FOUND, "比赛不存在");
        }

        // 如果没有指定 mode，使用比赛的类型
        String rankMode = mode != null ? mode.toUpperCase() : contest.type;

        if ("ACM".equals(rankMode)) {
            List<ContestAcmRankVO> ranks = acmRankService.getRankList(contestId);
            return ApiResponse.ok(ranks);
        } else if ("OI".equals(rankMode)) {
            List<ContestOiRankVO> ranks = oiRankService.getRankList(contestId);
            return ApiResponse.ok(ranks);
        } else {
            throw new BizException(ErrorCode.BAD_REQUEST, "无效的榜单模式");
        }
    }

    /**
     * 获取榜单快照（用户）
     * GET /api/v1/contests/{contestId}/scoreboard/snapshot/{type}
     */
    @GetMapping("/{contestId}/scoreboard/snapshot/{type}")
    public ApiResponse<ContestScoreboardSnapshot> getSnapshot(...) { ... }

    // ✅ 删除所有管理操作（已移到 AdminContestRankController）
}
```

**修复效果**:
- ✅ 用户接口在 `/api/v1/contests`

- ✅ 统一榜单接口 `GET /rank?mode=ACM/OI`

## 榜单系统架构

### 修复前的架构（双系统）

```
提交判题完成
    ↓
JudgePollingService.pollOne()
    ↓
    ├─ updateContestStanding()  ← 旧系统
    │   └─ contest_standings 表
    │       ├─ solved (AC 题数)
    │       ├─ penalty (罚时)
    │       ├─ score (总分)
    │       └─ last_ac_time
    │
    └─ ❌ 新系统未被调用
        ├─ contest_acm_rank_cache
        ├─ contest_acm_rank_problems
        ├─ contest_oi_rank_cache
        └─ contest_oi_rank_problems
```

### 修复后的架构（统一系统）

```
提交判题完成
    ↓
JudgePollingService.pollOne()
    ↓
updateContestRank()
    ↓
    ├─ ACM 赛制 → ContestAcmRankService.updateRankAfterJudge()
    │   ├─ contest_acm_rank_cache (总榜)
    │   │   ├─ rank_no (排名)
    │   │   ├─ solved_count (AC 题数)
    │   │   ├─ penalty_time (罚时)
    │   │   ├─ submission_count (提交次数)
    │   │   └─ last_ac_time (最后 AC 时间)
    │   │
    │   └─ contest_acm_rank_problems (单题状态)
    │       ├─ is_solved (是否 AC)
    │       ├─ wrong_attempts (错误次数)
    │       ├─ solve_time_minutes (AC 耗时)
    │       └─ first_ac_time (首次 AC 时间)
    │
    └─ OI 赛制 → ContestOiRankService.updateRankAfterJudge()
        ├─ contest_oi_rank_cache (总榜)
        │   ├─ rank_no (排名)
        │   ├─ total_score (总分)
        │   ├─ solved_count (满分题数)
        │   ├─ submission_count (提交次数)
        │   └─ last_score_update_time (最后得分时间)
        │
        └─ contest_oi_rank_problems (单题分数)
            ├─ best_score (最高分)
            ├─ full_score (满分值)
            ├─ best_submission_id (最佳提交)
            └─ first_full_score_time (首次满分时间)
```

## 榜单系统特性

### ACM 赛制特性

1. ✅ **AC 题数优先** - 排序：solved DESC
2. ✅ **罚时排序** - 次要排序：penalty ASC
3. ✅ **最后 AC 时间** - 第三排序：last_ac_time ASC
4. ✅ **错误罚时** - 每次错误提交罚 20 分钟（可配置）
5. ✅ **AC 耗时** - 从比赛开始到 AC 的分钟数
6. ✅ **幂等性** - 重复判题不会重复计分
7. ✅ **首次 AC 追踪** - 记录每题的首次 AC 时间

### OI 赛制特性

1. ✅ **总分优先** - 排序：total_score DESC
2. ✅ **满分题数** - 次要排序：solved_count DESC
3. ✅ **最后得分时间** - 第三排序：last_score_update_time ASC
4. ✅ **最高分机制** - 只保留最高分提交
5. ✅ **满分追踪** - 记录首次满分时间
6. ✅ **幂等性** - 重复判题不会错误更新
7. ✅ **部分分支持** - 每题有独立的满分值

### 通用特性

1. ✅ **实时更新** - 判题完成立即更新榜单
2. ✅ **全量重建** - 管理员可以重建整个榜单
3. ✅ **榜单快照** - 支持冻结榜、最终榜、自定义快照
4. ✅ **提交计数** - 追踪每个参赛者的提交次数
5. ✅ **单题明细** - 保留每题的详细状态
