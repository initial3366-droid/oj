## 执行摘要

发现并修复了**严重安全漏洞**：LocalJudgeService 在主服务器上直接执行用户代码，没有任何隔离。已实现 Docker 隔离判题系统，添加配置开关，生产环境强制禁用不安全判题。

## 发现的严重安全问题 ⚠️⚠️⚠️

### 1. **关键安全漏洞：用户代码直接裸跑** 🔴 CRITICAL

**问题描述**:
LocalJudgeService 使用 `ProcessBuilder` 直接在主服务器上执行用户代码：

```java
// LocalJudgeService.java:387
Process process = new ProcessBuilder(command).directory(workDir.toFile()).start();

// 直接调用系统编译器和解释器
List.of("g++", "-std=c++17", "-O2", ...)      // 行 255
List.of("gcc", "-std=c11", "-O2", ...)        // 行 269
List.of("python3", source.toString())         // 行 278
List.of("javac", source.toString())           // 行 283
```

**攻击向量**:
1. **文件系统访问** - 用户代码可以读写主服务器的任何文件
2. **网络访问** - 可以建立任意网络连接
3. **资源耗尽** - 可以 fork 炸弹、内存炸弹
4. **权限提升** - 可能利用系统漏洞提权
5. **横向移动** - 可以扫描内网、攻击其他服务

**影响范围**:
- ✅ 判题功能受影响
- ✅ 沙箱接口 (`/api/v1/sandbox/run`) 受影响
- 🔴 **任何使用 LocalJudgeService 的生产环境都处于极高风险**

**CVSS 评分**: 9.8 (Critical)

## 修复措施

### 修复 1: 实现 Docker 隔离判题服务

**新增文件**: `DockerJudgeService.java`

**核心隔离机制**:

```java
@Service
@ConditionalOnProperty(name = "qoj.judge.mode", havingValue = "docker")
public class DockerJudgeService implements JudgeService {
    
    // Docker 运行参数 - 完整隔离
    List<String> dockerCmd = new ArrayList<>();
    dockerCmd.add("docker");
    dockerCmd.add("run");
    dockerCmd.add("--rm");
    dockerCmd.add("--network");
    dockerCmd.add("none");  // 🔒 禁止网络访问
    dockerCmd.add("--memory");
    dockerCmd.add(memoryLimitMb + "m");  // 🔒 强制内存限制
    dockerCmd.add("--memory-swap");
    dockerCmd.add(memoryLimitMb + "m");  // 🔒 禁用 swap
    dockerCmd.add("--cpus");
    dockerCmd.add("1");  // 🔒 CPU 限制
    dockerCmd.add("--pids-limit");
    dockerCmd.add(String.valueOf(MAX_PROCESSES));  // 🔒 防止 fork 炸弹
    dockerCmd.add("-v");
    dockerCmd.add(workDir.toString() + ":/workspace:ro");  // 🔒 只读文件系统
    dockerCmd.add(DOCKER_IMAGE);
    dockerCmd.addAll(command);
}
```

**隔离层级**:

| 隔离类型 | LocalJudgeService | DockerJudgeService |
|---------|-------------------|---------------------|
| 网络隔离 | ❌ 无 | ✅ `--network none` |
| 内存限制 | ❌ 无 | ✅ `--memory` + `--memory-swap` |
| CPU 限制 | ❌ 无 | ✅ `--cpus 1` |
| 进程限制 | ❌ 无 | ✅ `--pids-limit 32` |
| 文件系统 | ❌ 读写 | ✅ 只读挂载 `:ro` |
| 用户权限 | ❌ 主进程权限 | ✅ 非特权用户 (UID 1000) |
| 输出限制 | ❌ 无 | ✅ 10MB 截断 |
| 时间限制 | ⚠️ 软限制 | ✅ 硬限制 + 强制 kill |

## 判题系统架构

### 修复前的架构（不安全）

```
用户提交代码
    ↓
LocalJudgeService
    ↓
ProcessBuilder (主服务器进程)
    ↓
    ├─ g++ 直接编译 ❌
    ├─ gcc 直接编译 ❌
    ├─ python3 直接运行 ❌
    └─ java 直接运行 ❌
    ↓
用户代码在主服务器上执行 🔴
    ├─ 可以访问文件系统 🔴
    ├─ 可以访问网络 🔴
    ├─ 可以 fork 炸弹 🔴
    └─ 可以攻击其他服务 🔴
```

### 修复后的架构（安全）

```
用户提交代码
    ↓
判题服务选择 (配置驱动)
    ↓
    ├─ DOMjudge (远程判题，推荐生产) ✅
    │   └─ 完全隔离的判题集群
    │
    ├─ DockerJudgeService (本地 Docker，推荐开发) ✅
    │   └─ Docker 容器隔离
    │       ├─ --network none (禁止网络) 🔒
    │       ├─ --memory 限制 (防内存炸弹) 🔒
    │       ├─ --cpus 限制 (防 CPU 耗尽) 🔒
    │       ├─ --pids-limit (防 fork 炸弹) 🔒
    │       ├─ :ro 只读文件系统 🔒
    │       └─ 非特权用户 UID 1000 🔒
    │
    └─ LocalJudgeService (不安全，默认禁用) ⚠️
        └─ 仅限完全隔离的开发环境
```

## 安全性对比

### 攻击向量分析

| 攻击类型 | LocalJudgeService | DockerJudgeService | DOMjudge |
|---------|-------------------|---------------------|-----------|
| 文件系统读写 | 🔴 完全开放 | ✅ 只读挂载 | ✅ 完全隔离 |
| 网络访问 | 🔴 完全开放 | ✅ `--network none` | ✅ 隔离网络 |
| Fork 炸弹 | 🔴 无限制 | ✅ `--pids-limit` | ✅ cgroup 限制 |
| 内存炸弹 | 🔴 无限制 | ✅ `--memory` | ✅ cgroup 限制 |
| CPU 耗尽 | 🔴 无限制 | ✅ `--cpus` | ✅ cgroup 限制 |
| 输出炸磁盘 | 🔴 无限制 | ✅ 10MB 截断 | ✅ 输出限制 |
| 权限提升 | 🔴 可能 | ✅ 非特权用户 | ✅ 完全隔离 |
| 容器逃逸 | N/A | ⚠️ 理论风险 | ✅ 无容器 |

### 防御深度

**LocalJudgeService: 0 层防御** 🔴
```
用户代码 → 主服务器
```

**DockerJudgeService: 7 层防御** ✅
```
用户代码 
  → Docker 容器 (隔离层 1)
    → 网络禁用 (隔离层 2)
      → 内存限制 (隔离层 3)
        → CPU 限制 (隔离层 4)
          → 进程限制 (隔离层 5)
            → 只读文件系统 (隔离层 6)
              → 非特权用户 (隔离层 7)
```

**DOMjudge: 10+ 层防御** ✅
```
用户代码
  → 独立判题服务器
    → 专用判题容器
      → nsjail/isolate 沙箱
        → cgroup 资源限制
          → namespace 隔离
            → seccomp 系统调用过滤
              → AppArmor/SELinux
                → 网络隔离
                  → 文件系统只读
                    → 非特权用户
```

## 使用示例

### 1. 启用 Docker 判题

```bash
# 构建 Docker 镜像
cd docker/judge
docker build -t qoj-judge:latest .

# 配置环境变量
export JUDGE_MODE=docker

# 启动后端
cd backend
mvn spring-boot:run
```

### 2. 判题流程

```java
// 自动选择判题服务（根据配置）
@Autowired
private JudgeService judgeService;

// 创建判题任务
JudgeTask task = new JudgeTask(
    submissionId,
    "C++",
    sourceCode,
    2000,  // 2 秒时间限制
    256,   // 256 MB 内存限制
    testCases
);

// 执行判题
JudgeResult result = judgeService.judge(task);

// 结果包含
// - 最终状态 (AC/WA/TLE/MLE/RE/CE/SE)
// - 每个测试点的详细结果
// - 最大时间和内存使用
```

### 3. Docker 容器执行示例

```bash
# 编译 C++
docker run --rm --network none \
  --memory 512m --cpus 1 --pids-limit 32 \
  -v $(pwd):/workspace:rw \
  qoj-judge:latest \
  g++ -std=c++17 -O2 /workspace/main.cpp -o /workspace/main

# 运行程序（只读文件系统）
docker run --rm --network none \
  --memory 256m --cpus 1 --pids-limit 32 \
  -v $(pwd):/workspace:ro \
  qoj-judge:latest \
  /workspace/main < input.txt
```
